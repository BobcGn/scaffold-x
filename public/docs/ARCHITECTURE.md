# Scaffold-X 架构设计文档 (MVP)

> 版本：v0.2.0 | 状态：MVP 阶段 | 最后更新：2026-07-08

## 1. 项目愿景

Scaffold-X 是一个开源教育与工程基建平台，致力于弥合在校学生（单文件 / 竞赛算法背景）与工业级生产开发（模块解耦、CI/CD、高可用）之间的认知鸿沟。

核心原则：**接口统一、实现下放、渐进式认知**。

---

## 2. 架构哲学：极简单体仓库 (Monorepo / All-in-One)

Scaffold-X 拒绝引入任何外部静态网页生成器（Quartz、Docusaurus、VitePress 等）来构建知识库。我们的选择是**彻底的极简主义**：

- 所有文档均为**纯 Markdown 文件**，不依赖任何特定的 callout 扩展语法。
- 文档的最终展示态 = **GitHub 的原生 Markdown 渲染**。
- 一个仓库内，同时承载三类职责完全不同的资产：**脚手架源码**、**CI 拦截配置**、**规范法典**。

### 2.1 为什么选择 All-in-One

| 对比维度 | 外部文档站 (Quartz 等) | 单体仓库纯 Markdown |
|----------|----------------------|-------------------|
| 构建复杂度 | 需额外构建流程、部署管道 | 零构建，推送即渲染 |
| 维护成本 | 文档站与代码仓库分离，易漂移 | 文档与代码同仓，始终同步 |
| 学生认知负担 | 需理解两个系统的协作关系 | 一个仓库 = 一切，所见即所得 |
| 可发现性 | 文档藏在独立域名下 | 文档就在代码旁边，路径清晰 |
| 离线可读性 | 依赖构建产物 | 任何 Markdown 阅读器即可浏览 |
| 贡献门槛 | 需学习文档站的主题/组件体系 | 会写 Markdown 就能贡献文档 |

### 2.2 设计约束

- **纯标准 Markdown**：仅使用 GitHub Flavored Markdown (GFM) 支持的语法。禁止使用任何需要插件渲染的扩展语法（如 `!!! note`、`:::tip` 等）。
- **零构建依赖**：文档不经过任何编译、转换或静态生成步骤。`git push` = 文档更新完成。
- **路径即导航**：目录结构本身即文档的导航体系，不需要额外的侧边栏配置文件。

## 3. 仓库目录结构：三类资产的优雅共存

这是 Scaffold-X 最核心的架构决策——在一个仓库内，通过清晰的目录划分，让三类职责完全不同的资产各安其位、互不干扰。

```
scaffold-x/
│
├── .github/                          # ── 第一类资产：自动化拦截 ──
│   ├── workflows/
│   │   └── ci.yml                    #   CI 流水线（物理拦截网）
│   ├── PULL_REQUEST_TEMPLATE.md      #   PR 模板（行为约束）
│   └── ISSUE_TEMPLATE/               #   Issue 模板（问题分类）
│
├── docs/                             # ── 第二类资产：规范法典 ──
│   ├── ARCHITECTURE.md               #   本文件 — 架构设计文档
│   ├── RULES.md                      #   代码规范与提交准则
│   ├── CI-CD-GUIDE.md               #   CI/CD 流水线说明
│   └── ONBOARDING.md                 #   新手上路指南
│
├── src/                              # ── 第三类资产：脚手架源码 ──
│   ├── App.jsx                       #   应用入口组件
│   ├── main.jsx                      #   Vite 启动入口
│   └── ...
│
├── public/                           #   静态资源
├── index.html                        #   Vite HTML 入口
├── package.json                      #   依赖与脚本定义
├── eslint.config.js                  #   ESLint 配置
├── vite.config.js                    #   Vite 构建配置
├── CONTRIBUTING.md                   #   贡献指南（根目录 = 最高可见性）
└── README.md                         #   项目门面
```

### 3.1 三类资产的职责边界

```
┌─────────────────────────────────────────────────────────────┐

│  │                  │  │                  │  │            │  │
│  │  "冷酷的监工"     │  │  "耐心的导师"     │  │  "待检验的  │  │
│  │                  │  │                  │  │   代码"     │  │
```

### 3.2 为什么这样划分

**`.github/` — 自动化拦截层**

这一层是"不可协商的"。它定义了所有参与者（包括项目维护者）都必须遵守的物理规则：
- CI 流水线在每次 PR 时自动执行，任何一步失败即阻止合并。
- PR 模板强制要求填写业务背景和自查清单。
- Issue 模板引导用户分类问题，减少无效沟通。

**`docs/` — 知识法典层**

这一层是"循循善诱的"。它解释规则背后的"为什么"，而不是简单地告诉"必须怎样"：
- `ARCHITECTURE.md` — 从全局视角理解系统设计。
- `RULES.md` — 每条规范的详细解释和示例。
- `CI-CD-GUIDE.md` — 当 CI 报错时，如何定位和修复问题。
- `ONBOARDING.md` — 第一次参与项目时的完整路径。

**`src/` — 脚手架源码层**

这一层是"被检验的对象"。它提供一个经过精心设计的起点代码，让学生在正确的轨道上开始编码。

### 3.3 文档的导航结构

由于不使用任何文档站生成器，目录结构本身即导航：

```
README.md                ← 入口：项目是什么、为什么存在
  └── docs/
      ├── ONBOARDING.md  ← 第一站：我该怎么开始？
      ├── ARCHITECTURE.md ← 第二站：系统整体是怎么设计的？
      ├── RULES.md       ← 第三站：有哪些规范必须遵守？
      └── CI-CD-GUIDE.md ← 第四站：CI 报错了怎么办？
  └── CONTRIBUTING.md    ← 实战：从本地到 PR 的完整流程
```

任何 Markdown 阅读器（包括 GitHub 网页、VS Code、Typora）都能完整呈现这套导航。

## 4. 核心设计模式：物理拦截 vs 知识呈现

这是 Scaffold-X 最关键的架构决策——将**强制约束**与**知识教育**彻底解耦，但共享同一个仓库。

### 4.1 职责分离原则

| 维度 | 物理拦截（冷酷的监工） | 知识呈现（耐心的导师） |
|------|----------------------|----------------------|
| 载体 | `.github/workflows/ci.yml` | `docs/` 目录下的 Markdown 文件 |
| 存储位置 | `.github/` 目录 | `docs/` 目录 |
| 行为 | **拒绝**不合规的代码合并 | **解释**为什么被拒绝 |
| 时机 | PR 提交时实时触发 | 用户被拦截后主动查阅 |
| 语气 | 机器式的、不可协商的 | 教学式的、循循善诱的 |
| 渲染方式 | GitHub Actions 日志输出 | GitHub 原生 Markdown 渲染 |
| 举例 | `ESLint error: no-unused-vars` | `docs/RULES.md` 第 4.2 节 |

### 4.2 错误导流：从 CI 日志到 Markdown 文档

这是两类资产之间的关键连接点。当 CI 拦截失败时，错误信息中包含指向 `docs/` 中具体章节的路径：

```bash
# CI 流水线中的错误输出示例
echo "::error title=ESLint 规范违规::检测到未使用变量 (no-unused-vars)。
请查阅：docs/RULES.md#42-变量命名与使用规范
修复后重新提交。"
```

学生看到报错后，直接在仓库内点击路径即可跳转到对应的规范文档——**零跳转成本，零认知负担**。

```
CI 报错                          规范文档
  │                                │
  │  "请查阅 docs/RULES.md#4.2"   │
  │──────────────────────────────►│
  │                                │  "未使用变量的常见原因：
  │                                │   1. 导入后忘记使用 → 删除 import
  │                                │   2. 临时调试代码 → 提交前清理
  │                                │   3. 重构后遗留 → 检查引用链"
  │                                │
  │  修正代码，重新提交              │
  │◄──────────────────────────────│
```

## 6. CI/CD 流水线架构

### 6.1 拦截规则矩阵

| 检查项 | 工具 | 触发条件 | 失败后果 |
|--------|------|----------|----------|
| 代码风格 | ESLint | PR + Push to main | ❌ 拒绝合并 |
| 构建验证 | Vite Build | PR + Push to main | ❌ 拒绝合并 |
| 类型检查 | TypeScript (tsc) | PR + Push to main | ❌ 拒绝合并 |
| 提交规范 | Commitlint | PR | ⚠️ 警告 (MVP) |
| 测试覆盖 | Vitest | PR + Push to main | ❌ 拒绝合并 (Phase 2) |

### 6.2 错误导流机制

CI 流水线失败时，在 GitHub Actions 日志中输出结构化错误信息，**直接指向仓库内的 Markdown 文档**：

```bash
# 示例：ESLint 失败时的输出
echo "::error title=ESLint 规范违规::检测到未使用变量 (no-unused-vars)。
请查阅：docs/RULES.md#42-变量命名与使用规范
修复后重新提交。"
```

这种结构化输出让每个错误都有对应的**仓库内文档入口**，形成闭环——不需要跳转到任何外部站点。

## 7. 技术栈选型决策

| 组件 | 选型 | 理由 |
|------|------|------|
| 前端框架 | React 19 | 生态最成熟，工业界需求量最大 |
| 构建工具 | Vite 8 | 开发体验极佳，ESM 原生支持 |
| 代码检查 | ESLint 10 (flat config) | 行业标准，规则可扩展 |
| CI/CD | GitHub Actions | 与代码托管零摩擦集成 |
| 文档方案 | 纯 Markdown + GitHub 原生渲染 | 零构建、零依赖、所见即所得 |
| 包管理 | npm | 学生群体普及率最高 |

**关于文档方案的决策说明**：我们刻意不使用 Quartz、Docusaurus、VitePress 等静态站点生成器。原因有三：

1. **降低认知门槛**：学生只需要会写 Markdown，不需要学习任何文档站框架。
2. **消除构建复杂度**：文档更新不需要额外的 CI/CD 流程，`git push` 即完成。
3. **保证同步性**：文档与代码同仓，永远不会出现"代码改了文档没更新"的情况。

| 阶段 | 目标 | 交付物 |

## 9. 非功能性约束

- **Node.js 版本**：CI 统一使用 LTS 版本（当前 v22）
- **包管理器锁文件**：必须提交 `package-lock.json`，禁止 `--no-save`
- **分支策略**：`main` 分支受保护，必须通过 PR 合并
- **提交信息**：遵循 Conventional Commits 规范（Phase 2 强制）
- **文档语法**：仅使用 GFM 标准语法，禁止扩展 Markdown 语法
- **文档位置**：所有规范文档存放在 `docs/` 目录或根目录，不得散落在源码目录中
